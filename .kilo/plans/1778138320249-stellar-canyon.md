
**Objective:** Produce a codebase audit that is evidence-based, repo-specific, and usable for product handover.

## Audit Rules

- Focus on observable evidence from the repository: code, config, tests, docs, and CI.
- Separate facts, risks, and recommendations.
- Flag only what can be supported by a file reference, command output, or clearly named code path.
- Prefer concrete severity labels: `critical`, `high`, `medium`, `low`.
- Treat missing automation, missing tests, and missing documentation as findings when they affect handover readiness.

## Audit Scope

Review these areas against the current repo state:

1. Architecture and module boundaries
2. Code quality and maintainability
3. Security, secrets, and access control
4. Testing, coverage, and CI/CD
5. Dependencies and package management
6. Performance and runtime efficiency
7. Documentation and onboarding readiness
8. Workflow and product-process alignment

## Repo-Specific Evidence To Check

- `package.json` scripts and dependency sets
- `README.md` and docs under `docs/`
- `.github/workflows/ci-cd.yml`
- Lint output from `npm run lint`
- Unit test output from `npm run test:run`
- Coverage output from `npm run test:coverage`
- E2E output from `npm run test:e2e`
- Security scan output from `npm audit --audit-level=moderate`
- Any auth, storage, secrets, or env handling code paths in `src/` and `supabase/`

## Report Template

### 1. Executive Summary

- Overall readiness: `ready`, `mostly ready`, `needs work`, or `not ready`
- Short summary of the highest-risk issues
- What blocks handover, if anything

### 2. Findings

For each finding, include:

- Severity
- Title
- Evidence
- Impact
- Recommendation

### 3. Pass/Fail Checklist

Use this exact checklist and mark each item `pass`, `fail`, or `partial`:

- Architecture is modular and boundaries are clear
- Naming, structure, and duplication are acceptable
- No hardcoded secrets, credentials, or unsafe env handling
- Auth and access control flows are defined and reviewed
- Lint passes
- Unit tests exist and run successfully
- Coverage is measured and meets the project target
- E2E tests cover critical user journeys
- CI runs lint, tests, build, and security checks
- Dependencies are current enough for handover
- Performance risks are identified for critical paths
- README and onboarding docs match the current workflow
- Commit and release workflow are traceable enough for audit purposes

### 4. Evidence Log

List the files, commands, and outputs used to support the audit.

### 5. Open Risks

List unresolved items that need owner confirmation or follow-up.

## Audit Checklist

- [ ] Architecture documented and modular
- [ ] Core modules and dependencies identified
- [ ] No secrets or credentials exposed in source or config
- [ ] Access control and auth flow reviewed
- [ ] Lint passes or failures are explained
- [ ] Unit test coverage is measured
- [ ] E2E coverage exists for critical paths
- [ ] CI pipeline reflects the repo’s actual checks
- [ ] Security scans are available or explicitly missing
- [ ] Dependencies are reviewed for risk
- [ ] README and onboarding docs are current
- [ ] Workflow artifacts support product handover

## Audit Instructions For The Reviewer

1. Start with the most concrete evidence available.
2. Note what was checked and what was not checked.
3. Do not mark a section as passing without evidence.
4. If a required check cannot be run, say why and record the gap.
5. Keep the report concise, factual, and ready to hand off.
