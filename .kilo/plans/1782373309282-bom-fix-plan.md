# Plan: Fix Runtime Error in `generate-single-post` / `encrypt-api-key`

## Status: RESOLVED ✅ — 2026-06-25

---

## Original Hypothesis (Incorrect)

The plan assumed the error at `lineno: 0, colno: 0` was caused by a **UTF-8 BOM** (Byte Order Mark) at the start of the file.

**This was wrong.** Both files were confirmed to have no BOM:

```
index.ts         — First 8 bytes: 0x64 0x65 0x63 0x6C 0x61 0x72 0x65 0x20  (= "declare ")  — No BOM
promptHelpers.ts — First 8 bytes: 0x64 0x65 0x63 0x6C 0x61 0x72 0x65 0x20  (= "declare ")  — No BOM
```

## Actual Root Cause

**The `dashboard-deploy/encrypt-api-key.ts` bundle was stale** — it was missing the entire `"validate"` action handler block (50 lines) that was added to `supabase/functions/encrypt-api-key/index.ts` in commit `7d8c6b5`.

### What happened:

1. Commit `7d8c6b5` added a `"validate"` action to `encrypt-api-key/index.ts` — this powers the "Test key" button in `ApiKeySettings.tsx` which calls `encrypt-api-key` with `{ action: "validate", apiKey, provider }`.

2. The `scripts/inline-edge-functions.mjs` script was **not re-run** after this change, so `dashboard-deploy/encrypt-api-key.ts` (the paste-ready bundle deployed to the Lovable Dashboard) did not include the new `validate` handler.

3. When a user clicked "Test key" in Profile → API Keys, the deployed function received `action: "validate"` but had no handler for it. It fell through directly to the **default "save" action** which expected `apiKey` and `provider` in the body — but the validate path passes `candidateKey` not `apiKey`. The Edge Function then returned `400: "Invalid request parameters."` or crashed at the `supabase.rpc("upsert_encrypted_api_key", ...)` call with a missing `apiKey` parameter.

4. This crash triggered a blank screen (`has_blank_screen: true`) because the client-side `validateUserApiKey()` in `apiKeyManager.ts` received an unexpected error response and threw, which propagated up to the Error Boundary.

5. The error appeared to come from `generate-single-post` in the error reporter because that function was being called **simultaneously** (the user was likely trying to generate content while also testing a key), or the error reporter attributed the crash to the wrong file.

### Why `lineno: 0, colno: 0`

This is characteristic of a **Supabase Edge Function runtime crash** (Deno `serve()` handler throwing an unhandled exception), not a syntax/BOM error. Deno's error reporting for unhandled promise rejections at the top level reports `0:0` when the stack trace cannot be mapped back to a source position.

---

## Fix Applied

### 1. Regenerated the stale bundle

Ran:
```bash
node scripts/inline-edge-functions.mjs
```

Result: `dashboard-deploy/encrypt-api-key.ts` now includes the 50-line `"validate"` action block that was missing. All other 16 bundles were unchanged (already up to date).

### 2. What the fix adds to the deployed bundle

The `"validate"` action handler (lines ~1829–1882 in the regenerated file):
- Receives `{ action: "validate", apiKey, provider }` 
- Makes a minimal live call (`max_tokens: 1`) to the provider via `callAI()`
- Returns `{ valid: true }` on success, or `{ valid: false, reason: "..." }` on failure
- Maps provider status codes (401, 402, 429, 500, 404) to human-readable reasons
- Never persists the key on this path

## Prevention: Always Re-run the Inline Script After Changes

The `README.md` in `dashboard-deploy/` and the script header both say:

> *"Re-run this whenever a function or _shared changes."*

This was not done for commit `7d8c6b5`. Going forward, any change to:
- `supabase/functions/*/index.ts`
- `supabase/functions/_shared/promptHelpers.ts`

...must be followed by:
```bash
node scripts/inline-edge-functions.mjs
```

And the relevant `dashboard-deploy/*.ts` file(s) must be re-deployed to the Supabase Dashboard.
